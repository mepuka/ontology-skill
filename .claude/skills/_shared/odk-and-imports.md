# ODK, Imports, and Module Management

**Referenced by:** `ontology-scout`, `ontology-architect`, `ontology-curator`,
`ontology-validator`, `ontology-mapper`.

Every ontology in this monorepo depends on upstream terms. This file covers
how to decide between standalone POD (Programmatic Ontology Development)
and ODK (Ontology Development Kit) for a project, how to pick an extraction
method (MIREOT / STAR / BOT / TOP), and how to keep imports fresh.

## 1. ODK vs standalone POD

### When to use ODK

- The project will join the OBO Foundry.
- The project has >3 upstream imports that change frequently.
- The team prefers a prescribed edit/release split with a Makefile-based
  CI pipeline.
- Downstream tools expect OBO-style artifacts (`-base`, `-full`, `-simple`).

### When to use standalone POD

- The project lives in this monorepo with `uv`-driven build scripts.
- The project is consumed by a specific downstream system (e.g.,
  `skygest-energy-vocab` for the Skygest platform).
- Imports are stable or refreshed on a project-driven cadence.
- The team wants direct Python-level control over every artifact.

Default for this monorepo: **standalone POD** using `ontologies/{name}/scripts/`
and `Justfile` targets. Record the decision in `docs/scope.md § Build regime`.

## 2. Non-ODK import management

For standalone POD projects, imports live under `ontologies/{name}/imports/`
as extracted modules. One file per upstream ontology.

### 2.1 Extraction methods

| Method | When to use | ROBOT command |
|--------|-------------|---------------|
| `MIREOT` | You want only the terms you name, minimal context. | `robot extract --method MIREOT --lower-terms terms.txt` |
| `STAR` | You want the terms plus their ancestors and property relations (good default). | `robot extract --method STAR --term-file terms.txt` |
| `BOT` | You want a bottom-up module: terms plus everything that depends on them. | `robot extract --method BOT --term-file terms.txt` |
| `TOP` | You want a top-down module: terms plus everything they depend on. | `robot extract --method TOP --term-file terms.txt` |
| `SUBSET` | You want just a list, no relations. | `robot extract --method SUBSET --term-file terms.txt` |

Default for this monorepo: **STAR** for schema-level imports, **MIREOT**
for targeted term reuse (e.g., grabbing five QUDT units).

### 2.2 Term file conventions

One term per line, CURIE form, with optional comment:

```
# QUDT quantity kinds relevant to generation assets
qudt:QK_Power
qudt:QK_Voltage
qudt:QK_Frequency

# Units
qudt:U_MegaW
qudt:U_KiloW
qudt:U_Volt
qudt:U_HZ
```

Store at `ontologies/{name}/imports/{upstream}-terms.txt`.

### 2.3 Extraction command pattern

```bash
.local/bin/robot extract \
  --input-iri http://qudt.org/2.1/vocab/unit \
  --method STAR \
  --term-file ontologies/{name}/imports/qudt-unit-terms.txt \
  --output ontologies/{name}/imports/qudt-unit.ttl
```

Prefer `--input-iri` (fetch upstream) over `--input file` (local cache)
for refreshes. Pin the upstream IRI with a version suffix when the
upstream offers one (e.g., `/2.1/`).

## 3. `imports-manifest.yaml`

Every ontology has `ontologies/{name}/imports/imports-manifest.yaml`
recording for each import:

```yaml
imports:
  - id: qudt-unit
    source: http://qudt.org/2.1/vocab/unit
    version_iri: http://qudt.org/2.1/vocab/unit
    retrieval_date: 2026-04-14
    extraction_method: STAR
    term_file: qudt-unit-terms.txt
    module_file: qudt-unit.ttl
    refresh_cadence_days: 90
    known_obsolete:
      - curie: qudt:U_PU_FAKE
        replaced_by: qudt:U_PER_UNIT
        noticed_on: 2026-03-12
  - id: bfo-core
    source: https://raw.githubusercontent.com/BFO-ontology/BFO/v2.0/bfo.owl
    version_iri: http://purl.obolibrary.org/obo/bfo/2020/bfo.owl
    retrieval_date: 2026-02-01
    extraction_method: SUBSET
    term_file: bfo-core-terms.txt
    module_file: bfo-core.ttl
    refresh_cadence_days: 365
```

Regenerated by `ontologies/{name}/scripts/refresh_imports.py` (or a
Justfile target). Safety rule 15 requires regeneration on any import
change.

## 4. Refresh cadence

| Ontology type | Typical cadence | Notes |
|--------------|-----------------|-------|
| Upper (BFO) | Yearly | Very stable. Reread the release notes. |
| Domain (QUDT, FOAF, DCAT) | Quarterly | Watch for deprecations and new subproperties. |
| Community (Schema.org) | Monthly | Schema.org has frequent additions but stable core. |
| Datacube | Yearly | Stable recommendation. |
| OEO / OBI / oeo-module | Quarterly | Living domain ontology. |

The refresh job:

1. Re-extract each module per `imports-manifest.yaml`.
2. Regenerate the manifest (updates `retrieval_date`, `version_iri`).
3. Run the CQ regression (`scripts/run_cq_tests.py`).
4. If any CQ breaks or any term is obsolete, raise a loopback
   (`stale_import`) to `ontology-curator` per
   [`iteration-loopbacks.md`](iteration-loopbacks.md).

## 5. Staleness detection

Even between scheduled refreshes, detect staleness with:

```bash
# For each import, check if upstream version IRI differs from manifest
uv run python scripts/check_import_staleness.py  # future helper
```

Signs of stale imports:

- `robot report` flags a term that the upstream has marked obsolete.
- A CQ that used to pass now returns zero rows.
- An upstream term's definition changes in a way that affects our genus-differentia.

## 6. ODK quick reference (for projects that do use ODK)

When a project IS ODK-managed:

- Edit file: `src/ontology/{name}-edit.owl`.
- Release via `run.sh make prepare_release`.
- Dynamic imports: `imports/{id}_import.owl` regenerated by ODK.
- Pattern files: `src/patterns/dosdp/*.yaml`.

Use ODK's dynamic-imports refresh rather than the manual helper in § 4.

## 7. Anti-patterns

- **Imports without manifest.** Then nobody knows which upstream version
  the ontology depends on.
- **Extraction without a term file.** Downloading the whole upstream
  bloats the repo and the reasoner's working set.
- **Copying terms across ontologies.** This shadows the upstream
  identifier. Always import.
- **Refreshing imports without rerunning CQs.** Silent breakage.
- **MIREOT when you need ancestors.** Queries on missing ancestors return
  zero rows; use STAR.
