# Academic Foundations for Personal Agent Ontology Design

## Purpose

This document surveys the academic literature on agent memory, cognitive
architectures, conversation modeling, and temporal reasoning that informs the
design of a personal agent ontology. For each paper or framework, we extract
key concepts requiring ontological representation, proposed class hierarchies,
relations between concepts, and what is directly reusable.

This complements the existing research documents:
- `research-memory-architectures.md` -- memory stores, control processes, and KR frameworks
- `research-ontology-design.md` -- ontology design patterns, standards alignment, SHACL/SPARQL

---

## 1. CoALA: Cognitive Architectures for Language Agents

**Paper:** Sumers, Yao, Narasimhan, and Griffiths (2023). "Cognitive
Architectures for Language Agents." arXiv:2309.02427. Published in
Transactions on Machine Learning Research (02/2024).

**Source:** [arXiv:2309.02427](https://arxiv.org/abs/2309.02427)

### Summary

CoALA proposes a systematic framework for designing and understanding LLM-based
agents by drawing on classical cognitive architecture research. It organizes
agents along three dimensions: information storage (memory), action space, and
decision-making procedure.

### Memory Types

CoALA defines a structured memory taxonomy:

- **Working Memory**: Capacity-limited, actively maintained information for the
  current decision cycle. Contains perceptual inputs, active knowledge
  (generated by reasoning or retrieved from long-term memory), and core
  information carried over from the previous cycle (e.g., active goals). Maps
  to the LLM's context window / scratchpad.

- **Long-Term Episodic Memory**: Records of past events and experiences.
  Answers questions like "What happened the last time I tried solution X?"
  Stores specific, time-situated traces.

- **Long-Term Semantic Memory**: Factual knowledge about the world. Contains
  generalizations abstracted from experience, e.g., "Birds can fly, except
  for ostriches."

- **Long-Term Procedural Memory**: Knowledge of how to perform tasks. May be
  embedded in the agent's code, learned policies, or the LLM's parameters
  themselves.

### Action Space

CoALA distinguishes two categories of action:

- **Internal Actions**: Memory retrieval, reasoning, and learning operations
  that manipulate the agent's own knowledge stores.

- **External Actions (Grounding Actions)**: Interactions with the physical
  world, digital interfaces, or humans.

### Decision Loop (Agent Cycle)

The decision procedure operates as an interactive loop:

1. **Retrieval**: Pull relevant information from long-term memories into
   working memory.
2. **Reasoning**: Propose and evaluate candidate actions (both internal and
   external) using the LLM as a reasoning engine over working memory.
3. **Action Selection**: Choose the best action and execute it.
4. **Observation**: Receive environmental feedback.
5. **Learning/Write-back**: Update long-term memories based on experience.

### Ontological Concepts to Extract

| Concept | Proposed Class | Relations |
|---------|---------------|-----------|
| Agent | `Agent` | `hasWorkingMemory`, `hasLongTermMemory` |
| Working Memory | `WorkingMemory` | `containsGoal`, `containsPercept`, `containsActiveKnowledge` |
| Episodic Memory | `EpisodicMemory` | `storesEpisode`, subClassOf `LongTermMemory` |
| Semantic Memory | `SemanticMemory` | `storesFact`, subClassOf `LongTermMemory` |
| Procedural Memory | `ProceduralMemory` | `storesSkill`, `storesPolicy`, subClassOf `LongTermMemory` |
| Decision Cycle | `DecisionCycle` | `hasPhase`, `triggersAction` |
| Internal Action | `InternalAction` | `operatesOn` memory, subClassOf `Action` |
| External Action | `ExternalAction` | `interactsWith` environment, subClassOf `Action` |
| Grounding Action | `GroundingAction` | synonym for `ExternalAction` |

### Reusability Assessment

CoALA's memory taxonomy is directly reusable as the top-level class hierarchy
for agent memory in our ontology. The working/episodic/semantic/procedural
distinction is well-grounded in both cognitive science and software
architecture. The internal/external action distinction maps cleanly to BFO's
process hierarchy.

## 2. Generative Agents: Interactive Simulacra of Human Behavior

**Paper:** Park, O'Brien, Cai, Morris, Liang, and Bernstein (2023).
"Generative Agents: Interactive Simulacra of Human Behavior." UIST 2023.
arXiv:2304.03442.

**Source:** [arXiv:2304.03442](https://arxiv.org/abs/2304.03442) |
[ACM DL](https://dl.acm.org/doi/10.1145/3586183.3606763)

### Summary

The Stanford Generative Agents paper demonstrates 25 LLM-powered agents
inhabiting a sandbox world (Smallville), each exhibiting believable individual
and emergent social behavior. The key contribution is an agent architecture with
three core components: memory stream, reflection, and planning.

### Memory Stream

Each agent maintains a **memory stream** -- a chronologically ordered database
of every observation, plan, and reflection expressed in natural language. Each
entry records:

- Natural language description of the event/observation
- Timestamp (creation time)
- Last access time (for recency computation)
- Importance score (integer, assigned by LLM at creation time)

### Retrieval Mechanism

Memory retrieval combines three scoring factors:

- **Recency**: Exponential decay function based on time since last access.
  More recently accessed memories score higher.
- **Importance**: A fixed score (1-10) assigned at memory creation. Mundane
  events (e.g., eating breakfast) score low; significant events (e.g., a
  breakup, a career change) score high.
- **Relevance**: Cosine similarity between the query embedding and the memory
  text embedding. Captures semantic relatedness to the current context.

Final retrieval score = weighted combination of all three factors, normalized
to [0,1].

### Reflection

Reflection is triggered when the sum of importance scores of recent memories
exceeds a threshold. The reflection process:

1. Queries the memory stream: "Given recent experiences, what are the 3 most
   salient high-level questions?"
2. For each question, retrieves relevant memories.
3. Synthesizes "insight" statements (higher-order abstractions).
4. Stores reflections back into the memory stream as new entries with their
   own importance scores.
5. Reflections can recursively trigger further reflections, creating a
   hierarchical tree of increasingly abstract insights.

### Planning

Planning operates hierarchically:

1. **Daily Plan**: A broad sketch of the day's agenda based on persona traits
   and yesterday's events.
2. **Hourly Schedule**: Decomposition of the daily plan into hour-long blocks.
3. **Fine-Grained Actions**: Each block decomposed into 5-15 minute action
   steps.
4. **Reactive Adjustment**: At each time step, agents can revise plans based
   on new observations (e.g., encountering another agent triggers
   conversation).

Plans are stored in the memory stream and participate in retrieval, allowing
the agent to consider observations, reflections, and plans together when
deciding behavior.

### Ontological Concepts to Extract

| Concept | Proposed Class | Relations |
|---------|---------------|-----------|
| Memory Stream | `MemoryStream` | `containsEntry`, `ownedBy` Agent |
| Memory Entry | `MemoryEntry` | `hasTimestamp`, `hasImportanceScore`, `hasLastAccessTime` |
| Observation | `Observation` | subClassOf `MemoryEntry`, `observes` Entity/Event |
| Reflection | `Reflection` | subClassOf `MemoryEntry`, `derivedFrom` MemoryEntry+ |
| Plan | `Plan` | subClassOf `MemoryEntry`, `hasSubPlan`, `achievesGoal` |
| Daily Plan | `DailyPlan` | subClassOf `Plan` |
| Hourly Plan | `HourlyPlan` | subClassOf `Plan`, `partOf` DailyPlan |
| Action Plan | `ActionPlan` | subClassOf `Plan`, `partOf` HourlyPlan |
| Retrieval Score | `RetrievalScore` | `recencyComponent`, `importanceComponent`, `relevanceComponent` |
| Importance Score | (datatype property) | `xsd:integer`, range [1,10] |

### Reusability Assessment

The memory stream is essentially an append-only episodic memory with rich
metadata. The retrieval scoring mechanism (recency + importance + relevance)
is a concrete implementation of multi-factor memory access that our ontology
should support as a pattern. The reflection mechanism (experience to
abstraction) represents a consolidation process that bridges episodic to
semantic memory. The hierarchical planning decomposition is directly modelable
with part-of relations.

## 3. MemGPT: Towards LLMs as Operating Systems

**Paper:** Packer, Fang, Patil, Lin, Wooders, and Gonzalez (2023). "MemGPT:
Towards LLMs as Operating Systems." arXiv:2310.08560.

**Source:** [arXiv:2310.08560](https://arxiv.org/abs/2310.08560) |
[research.memgpt.ai](https://research.memgpt.ai/)

### Summary

MemGPT draws an analogy between OS virtual memory management and LLM context
management. Just as operating systems provide the illusion of large memory
through paging between RAM and disk, MemGPT provides the illusion of
unbounded context through intelligent management of a tiered memory hierarchy.

### Memory Architecture

**Main Context** (analogous to RAM) -- the LLM's prompt buffer, subdivided into:

- **System Instructions** (read-only): Static information about MemGPT control
  flow, memory level usage instructions, and function calling conventions.

- **Working Context** (read/write): A fixed-size block of unstructured text,
  writable only via MemGPT function calls. Stores essential facts and
  persistent state about the user/conversation.

- **FIFO Queue**: Rolling history of messages (user messages, agent messages,
  system messages, function call inputs/outputs). The first index stores a
  recursive summary of evicted messages.

**External Context** (analogous to disk storage):

- **Recall Storage**: A searchable database of all past messages. Supports
  semantic search for reconstruction of specific past conversations.

- **Archival Storage**: Long-term storage for important information. Supports
  arbitrary-length text objects and semantic search. Information can be moved
  back into main context as needed.

### Memory Management Operations

- **Page Out (Eviction)**: When prompt tokens exceed a warning threshold
  (~70% of context window), the system inserts a "memory pressure" warning.
  When they exceed a flush threshold (~100%), the queue manager evicts
  messages, generates a recursive summary, and stores evicted messages in
  recall storage.

- **Page In (Retrieval)**: When the agent calls a retrieval function, messages
  from recall or archival storage are appended to the FIFO queue, reinserting
  them into context.

- **Self-Editing**: The LLM can actively manage its own memory by deciding
  what to store, summarize, or move between tiers via tool/function calls.
  This is the key innovation -- the agent itself controls memory management.

### Ontological Concepts to Extract

| Concept | Proposed Class | Relations |
|---------|---------------|-----------|
| Memory Tier | `MemoryTier` | `hasCapacity`, `hasPersistence` |
| Main Context | `MainContext` | subClassOf `MemoryTier`, `contains` WorkingContext, FIFOQueue |
| Working Context | `WorkingContext` | `storesStateFact`, `hasMaxSize` |
| FIFO Queue | `MessageQueue` | `containsMessage`, `hasSummary` |
| External Context | `ExternalContext` | subClassOf `MemoryTier` |
| Recall Storage | `RecallStorage` | subClassOf `ExternalContext`, `supportsSemanticSearch` |
| Archival Storage | `ArchivalStorage` | subClassOf `ExternalContext`, `supportsSemanticSearch` |
| Memory Pressure Event | `MemoryPressureEvent` | `triggeredBy` threshold, `resultsIn` eviction |
| Eviction | `MemoryEviction` | `evictsFrom` MainContext, `movesTo` RecallStorage |
| Memory Retrieval | `MemoryRetrieval` | `retrievesFrom` ExternalContext, `insertsInto` MainContext |
| Recursive Summary | `RecursiveSummary` | `summarizes` Message+, `replacesInQueue` |

### Reusability Assessment

MemGPT's tiered memory model is directly relevant for modeling how actual LLM
agents manage context. The main context / external context distinction maps to
working memory / long-term memory. The self-editing mechanism (agent manages
its own memory via function calls) is a novel pattern that our ontology should
represent -- memory management as a type of internal action. The eviction and
retrieval operations are concrete instances of consolidation and recall.

## 4. Agent Memory Surveys (2024-2025)

### 4a. "A Survey on the Memory Mechanism of Large Language Model based Agents"

**Paper:** Zeyu Zhang et al. (2024). arXiv:2404.13501. Published in ACM
Transactions on Information Systems.

**Source:** [arXiv:2404.13501](https://arxiv.org/abs/2404.13501)

This survey organizes memory from three dimensions:

- **Memory Sources**: Where memory content comes from (interaction history,
  external knowledge, agent-generated summaries).
- **Memory Forms**: How memory is stored (natural language text, embeddings,
  structured data/KG, parametric/in-weights).
- **Memory Operations**:
  - **Read (Retrieval)**: Selecting relevant memories for current context.
  - **Write (Encoding)**: Storing new information from experience.
  - **Manage (Consolidation/Forgetting)**: Reorganizing, compressing,
    pruning, or deleting memories over time.

### 4b. "From Human Memory to AI Memory" Survey

**Paper:** Wu et al. (2025). "From Human Memory to AI Memory: A Survey on
Memory Mechanisms in the Era of LLMs." arXiv:2504.15965.

**Source:** [arXiv:2504.15965](https://arxiv.org/abs/2504.15965)

Proposes an **eight-quadrant classification framework** along three dimensions:

1. **Object dimension**: Personal memory vs System memory
   - Personal: user-specific information and interaction history
   - System: AI's internal processes, intermediate results
2. **Form dimension**: Parametric memory vs Non-parametric memory
   - Parametric: encoded in model weights
   - Non-parametric: external stores (text, vectors, KG)
3. **Time dimension**: Short-term memory vs Long-term memory
   - Short-term: current session/context
   - Long-term: persisted across sessions

### 4c. A-Mem: Agentic Memory for LLM Agents

**Paper:** Xu, Liang, et al. (2025). "A-MEM: Agentic Memory for LLM Agents."
NeurIPS 2025. arXiv:2502.12110.

**Source:** [arXiv:2502.12110](https://arxiv.org/abs/2502.12110)

A-Mem introduces the **Zettelkasten method** for agent memory: each memory is
stored as a structured note with contextual descriptions, keywords, and tags.
When a new memory is added, the system retrieves relevant historical memories
and uses an LLM to determine whether links should be established. This creates
an interconnected knowledge network with dynamic indexing.

Key design features:
- Each note has structured attributes (context, keywords, tags)
- Link generation connects related memories
- The network evolves through agent-driven organization
- Memories can be refined and updated over time

### Ontological Concepts from Surveys

| Concept | Proposed Class | Relations |
|---------|---------------|-----------|
| Memory Operation | `MemoryOperation` | `operatesOn` MemoryStore |
| Read Operation | `MemoryRead` | subClassOf `MemoryOperation` |
| Write Operation | `MemoryWrite` | subClassOf `MemoryOperation` |
| Manage Operation | `MemoryManage` | subClassOf `MemoryOperation` |
| Consolidation | `Consolidation` | subClassOf `MemoryManage` |
| Forgetting | `Forgetting` | subClassOf `MemoryManage` |
| Memory Form | `MemoryForm` | enumeration: text, embedding, structured, parametric |
| Memory Note (A-Mem) | `MemoryNote` | `hasKeyword`, `hasTag`, `linkedTo` MemoryNote |
| Personal Memory | `PersonalMemory` | `belongsTo` Agent |
| System Memory | `SystemMemory` | `usedBy` Agent |

### Reusability Assessment

The three-operation framework (read/write/manage) provides a clean action
taxonomy for memory operations. The eight-quadrant framework offers a useful
classification grid. A-Mem's structured note concept with inter-linking is
relevant for modeling how semantic memory items relate to each other. The
personal/system memory distinction is important for our agent ontology.

## 5. Classical Cognitive Architecture Foundations

### 5a. ACT-R (Adaptive Control of Thought -- Rational)

**Developer:** John R. Anderson, Carnegie Mellon University.

**Sources:** [ACT-R Official](https://act-r.psy.cmu.edu/about/) |
[Wikipedia](https://en.wikipedia.org/wiki/ACT-R)

ACT-R is a hybrid cognitive architecture combining a symbolic production
system with subsymbolic statistical processes.

**Memory Systems:**

- **Declarative Memory**: Stores facts as "chunks" -- symbolic structures
  with typed slots (attributes and values). Chunks represent declarative
  knowledge ("Washington, D.C. is the capital of the United States").
  Retrieval is governed by activation levels (base-level activation + spreading
  activation from current context).

- **Procedural Memory**: Stores skills as "production rules" -- if-then rules
  where the condition (if) specifies a pattern to match against buffer contents
  and the action (then) specifies buffer modifications. Only one production
  can fire per cycle (~50ms).

**Module-Buffer Architecture:**

ACT-R organizes cognition around specialized modules, each with an associated
buffer:

- **Goal Module/Buffer**: Tracks current goals and intentions.
- **Imaginal Module/Buffer**: Holds context information and intermediate
  problem representations.
- **Retrieval Buffer**: Interface to declarative memory.
- **Visual Module/Buffer**: Visual perception.
- **Manual Module/Buffer**: Motor control.

Cognition unfolds as a series of production firings, each matching buffer
contents and modifying them. The cognitive cycle takes approximately 50ms.

**Learning Mechanisms:**

- Declarative: Chunks are strengthened when accessed (base-level learning).
- Procedural: Production compilation creates new rules; utility learning
  adjusts rule selection.

**Ontological Mapping:**

| ACT-R Concept | Ontological Category (BFO-aligned) | Notes |
|---------------|-----------------------------------|-------|
| Chunk | `InformationContentEntity` (BFO: generically dependent continuant) | A structured fact representation |
| Production Rule | `Plan` or `Procedure` (BFO: generically dependent continuant) | Condition-action pattern |
| Buffer | `WorkingMemoryComponent` | Capacity-limited interface to a module |
| Module | `CognitiveModule` | Specialized processing unit |
| Activation Level | `DispositionMetric` (BFO: quality) | Subsymbolic accessibility measure |
| Production Firing | `CognitiveCycleStep` (BFO: process) | ~50ms atomic processing step |

### 5b. Soar

**Developer:** John E. Laird, University of Michigan.

**Sources:** [Soar Home](https://soar.eecs.umich.edu/home/About/) |
[arXiv:2205.03854](https://arxiv.org/pdf/2205.03854)

Soar is a general cognitive architecture designed for creating systems with
the same cognitive abilities as humans, emphasizing general AI functionality
and efficiency.

**Memory Systems:**

- **Working Memory**: The agent's current situation model. Stores goals,
  perceptual input, and intermediate reasoning results. All behavior is driven
  by interaction between working memory contents and procedural memory.

- **Procedural Memory (Long-Term)**: A production system whose rules apply to
  working memory. Learning via "chunking" (creating new rules from
  problem-solving traces) and reinforcement learning for operator selection.

- **Semantic Memory (SMEM, Long-Term)**: A very large store of fact-like
  structures represented as directed cyclic graphs. Encodes general knowledge
  the agent "knows" about itself and the world.

- **Episodic Memory (EPMEM, Long-Term)**: Automatically records temporally
  ordered snapshots of working memory. Supports sequential playthrough of
  past episodes, retrieval of specific memories by query, and search for
  episodes containing specific structures.

**Decision Cycle:**

Soar's decision cycle consists of:
1. Input phase (perception)
2. Elaboration (rule firing to expand working memory)
3. Decision (operator selection)
4. Application (operator execution)
5. Output (motor actions)

When the agent reaches an "impasse" (cannot decide), it creates a subgoal
and reasons about the impasse, potentially using episodic or semantic memory.

**Ontological Mapping:**

| Soar Concept | Ontological Category | Notes |
|-------------|---------------------|-------|
| Working Memory Element | `ActiveRepresentation` | Currently active knowledge |
| Production | `ProceduralKnowledge` | If-then rule |
| Chunking | `ProceduralLearning` | Compiling experience into rules |
| Semantic Memory Fact | `SemanticKnowledgeItem` | Graph-structured fact |
| Episodic Snapshot | `EpisodicRecord` | Time-stamped WM snapshot |
| Operator | `CognitiveAction` | Unit of deliberate behavior |
| Impasse | `DecisionFailure` | Triggers subgoal creation |

### 5c. LIDA (Learning Intelligent Decision Agent)

**Developer:** Stan Franklin, University of Memphis.

**Sources:** [Wikipedia](https://en.wikipedia.org/wiki/LIDA_(cognitive_architecture)) |
[CCRG Tutorial](https://ccrg.cs.memphis.edu/assets/framework/The-LIDA-Tutorial.pdf)

LIDA is grounded in Global Workspace Theory (GWT), implementing consciousness
as a computational mechanism for information integration and broadcast.

**Cognitive Cycle (Three Phases):**

1. **Understanding/Perception Phase** (80-100ms from stimulus):
   - Sensory stimuli are processed by perception.
   - Perceptual Associative Memory recognizes patterns.
   - Episodic and declarative memories are cued by percepts.
   - A "Current Situational Model" is built in the workspace.

2. **Attending/Consciousness Phase** (200-280ms from stimulus):
   - Attention codelets (specialized mini-agents) form coalitions around
     salient workspace content.
   - Coalitions compete for attention.
   - The winning coalition enters the Global Workspace.
   - Its contents are **broadcast globally** to all modules -- this is the
     "conscious broadcast."

3. **Action Selection Phase** (60-110ms from broadcast):
   - The broadcast triggers learning across all memory types.
   - Procedural memory instantiates possible action schemes.
   - Action selection chooses the behavior for this cycle.

**Memory Systems:**

- **Sensory Memory**: Very brief retention of raw sensory data.
- **Sensory-Motor Memory**: Learned associations between percepts and actions.
- **Perceptual Associative Memory (PAM)**: Recognizes objects/situations
  (implemented as a "slipnet" -- spreading activation network).
- **Transient Episodic Memory (TEM)**: Short-lived episodic records of
  conscious broadcasts. Automatically encodes what enters consciousness.
- **Declarative Memory**: Longer-term consolidated memories (moved from TEM
  through consolidation if not decayed).
- **Procedural Memory**: Action schemes (implemented as a "scheme net").

**Key Concept -- Codelets:**

LIDA relies on "codelets" -- special-purpose, relatively independent
mini-agents, each implemented as a small piece of code running as a separate
thread. Types include attention codelets, perception codelets, behavior
codelets, and learning codelets.

**Ontological Mapping:**

| LIDA Concept | Ontological Category | Notes |
|-------------|---------------------|-------|
| Cognitive Cycle | `CognitiveCycle` (BFO: process) | ~300ms understand-attend-act |
| Global Workspace | `ConsciousBroadcast` | Central integration point |
| Codelet | `MiniAgent` or `CognitiveProcess` | Specialized processor |
| Attention Codelet | `AttentionProcess` | Selects salient content |
| Conscious Broadcast | `InformationBroadcastEvent` | Makes info globally available |
| Transient Episodic Memory | `ShortTermEpisodicStore` | Auto-records conscious content |
| Consolidation | `MemoryConsolidation` | TEM to declarative transfer |
| Coalition | `AttentionCoalition` | Competing groups of content |

### Cross-Architecture Comparison for Ontology Design

| Feature | ACT-R | Soar | LIDA | Implication for Ontology |
|---------|-------|------|------|------------------------|
| Working Memory | Buffers (capacity-limited) | Graph structures (unbounded) | Workspace + coalitions | Model as capacity-limited with explicit components |
| Episodic Memory | Not native (added later) | EPMEM snapshots | Transient + Consolidated | Model with temporal ordering and decay/consolidation |
| Semantic Memory | Declarative chunks | SMEM (cyclic graphs) | Declarative Memory | Model as structured facts with activation/accessibility |
| Procedural Memory | Production rules | Productions + chunking | Scheme nets | Model as condition-action patterns |
| Learning | Activation strengthening | Chunking + RL | Via conscious broadcast | Model as memory state transitions |
| Control | Production cycle (~50ms) | Decision cycle | Cognitive cycle (~300ms) | Model as repeating process with phases |
| Consciousness | Not explicit | Not explicit | Global Workspace broadcast | Optional: model as attention/broadcast mechanism |

### Reusability Assessment

All three architectures converge on a common memory taxonomy:
working/episodic/semantic/procedural. This convergence strongly supports
adopting this as our primary memory classification. The key differences
(ACT-R's activation-based retrieval, Soar's impasse-driven learning, LIDA's
broadcast-based integration) suggest that memory access mechanisms should be
modeled as configurable properties rather than fixed classes. The cognitive
cycle concept from all three architectures maps to the CoALA decision loop
and should be a core process class.

## 6. BDI (Belief-Desire-Intention) Models

### Philosophical Foundations: Bratman (1987)

**Work:** Bratman, Michael E. (1987). *Intention, Plans, and Practical
Reasoning.* Harvard University Press.

**Source:** [PhilPapers](https://philpapers.org/rec/BRAIPA)

Bratman argues that intentions are a **third elementary mental state**,
irreducible to beliefs and desires. Key claims:

- **Future-directed intentions** are distinct from present-directed intentional
  action. They are commitments to future courses of action that constrain
  further practical reasoning.

- **Intentions as plan elements**: Intentions are elements of partial plans
  that structure activity over time. Plans are hierarchical -- high-level
  intentions are filled in with more specific sub-plans through means-end
  reasoning.

- **Rational constraints on intentions**:
  - **Consistency**: Intentions must be consistent with beliefs.
  - **Means-end coherence**: If you intend an end, you should intend the
    necessary means (or at least deliberate about them).
  - **Stability**: Intentions resist arbitrary reconsideration (but are not
    irrevocable).

- **Three roles of intention**:
  1. Terminate deliberation (commitment function)
  2. Provide input to further practical reasoning (means-end)
  3. Coordinate activity over time and with others (social coordination)

### BDI Software Model

**Sources:** [Wikipedia](https://en.wikipedia.org/wiki/Belief%E2%80%93desire%E2%80%93intention_software_model) |
[Georgeff & Pell (1998)](https://link.springer.com/chapter/10.1007/3-540-49057-4_1)

The BDI software model implements Bratman's theory for artificial agents:

- **Beliefs**: The agent's informational state -- what it holds to be true
  about the world. Updated through perception and inference.

- **Desires**: The agent's motivational state -- states of affairs it would
  like to achieve. May be mutually inconsistent (wishful thinking allowed).

- **Intentions**: The agent's deliberative state -- desires that the agent has
  committed to pursuing. Must be internally consistent and consistent with
  beliefs. Drive plan selection and execution.

The BDI control loop:
1. Perceive environment and update beliefs.
2. Generate options (desires/goals) based on current beliefs and intentions.
3. Filter options to select intentions (deliberation).
4. Execute plans to achieve intentions.
5. Monitor execution and revise as needed.

### AgentSpeak/Jason and OWL Integration

AgentSpeak is a logic programming language for BDI agents. Jason is its most
prominent implementation. Recent work on JASDL (Jason + Semantic Description
Logic) integrates ontological reasoning via the OWL API, enabling:

- More expressive queries to the belief base using ontological inference
- Belief updates verified for ontological consistency
- Knowledge sharing between agents via OWL serialization

### The BDI Ontology (2024-2025)

**Paper:** "The Belief-Desire-Intention Ontology for modelling mental reality
and agency." arXiv:2511.17162.

**Source:** [arXiv:2511.17162](https://arxiv.org/abs/2511.17162)

A recent formal BDI Ontology has been developed as a **modular Ontology Design
Pattern (ODP)** with the following features:

- Captures beliefs, desires, intentions, goals, plans, and justifications
  with their dynamic interrelations.
- Aligns with DOLCE+DnS UltraLite (DUL) foundational ontology.
- Supports the **Triples-to-Beliefs-to-Triples (T2B2T)** paradigm for
  bidirectional flow between RDF triples and mental states.
- Standard OWL serialization (RDF/XML and Turtle).
- Distinguishes **dispositional** and **occurrent** mental states:
  - Beliefs and desires have dual nature (dispositional = standing tendency;
    occurrent = currently active/experienced).
  - Intentions are dispositions to act that emerge from deliberation between
    occurrent beliefs and occurrent desires.

| Intention | `Intention` | `heldBy` Agent, `committsTo` Plan, `achievesGoal` Goal |
| Goal | `Goal` | `motivatedBy` Desire, `addressedBy` Plan |
| Plan | `Plan` | `hasStep` Action, `achieves` Goal, `hasSubPlan` Plan |

### Reusability Assessment

The BDI framework is **highly reusable** for our personal agent ontology.
Beliefs, desires, and intentions map directly to the epistemic and
goal-oriented aspects of agent memory. The existing BDI Ontology (ODP) aligned
with DUL is directly importable. The dispositional/occurrent distinction is
important for representing standing knowledge vs. currently active mental
states. The T2B2T paradigm is relevant for bridging our RDF knowledge graph
with agent reasoning processes.

## 7. Conversation and Dialog Modeling

### 7a. DAMSL: Dialog Act Markup in Several Layers

**Authors:** Core and Allen (1997).

**Sources:** [Rochester CS](https://www.cs.rochester.edu/research/speech/damsl/RevisedManual/) |
[Summary PDF](https://staff.fnwi.uva.nl/r.fernandezrovira/teaching/upf-materials/Summary_DAMSL.pdf)

DAMSL provides a multi-layered annotation scheme for dialog acts. Four tag sets:

1. **Communicative Status**: Whether the utterance is intelligible, a
   self-talk, etc.

2. **Information Level**: Task vs. task-management vs. communication-management.

3. **Forward-Looking Functions** (what the utterance does going forward):
   - **Statement (Assert)**: Makes claims about the world (can be evaluated
     true/false).
   - **Info-Request**: Seeks information.
   - **Influencing-Addressee-Future-Action (Directive)**: Requests, suggests,
     or commands action.
   - **Committing-Speaker-Future-Action (Commissive)**: Promises, offers,
     or commits to action.
   - **Open Option**: Presents options without directing.

4. **Backward-Looking Functions** (how it relates to prior discourse):
   - **Agreement**: Accept, reject, hold.
   - **Understanding**: Signal understanding, misunderstanding, request
     clarification.
   - **Answer**: Provides requested information.
   - **Information-Relation**: Relates to prior content (correction,
     elaboration).

Key design principle: **multiple tags per utterance** -- a single utterance
can simultaneously answer a question, confirm understanding, and commit to
an action.

### 7b. DIT++ and ISO 24617-2

**Developer:** Harry Bunt, Tilburg University.

**Sources:** [DIT++ site](https://dit.uvt.nl/) |
[ISO 24617-2 (2020)](https://aclanthology.org/2020.lrec-1.69.pdf)

DIT++ extends DAMSL into a comprehensive multidimensional taxonomy that became
the basis for the ISO 24617-2 standard for dialogue act annotation.

**Ten Orthogonal Dimensions:**

1. **Task**: Domain-specific communicative functions
2. **Auto-Feedback**: Speaker's processing of own utterance
3. **Allo-Feedback**: Speaker's processing of partner's utterance
4. **Turn Management**: Turn allocation (take, give, keep)
5. **Time Management**: Requesting/granting time
6. **Discourse Structuring**: Topic management, opening/closing
7. **Own Communication Management**: Self-correction, rephrasing
8. **Partner Communication Management**: Completion, correction of partner
9. **Social Obligations Management**: Greetings, apologies, thanks
10. **Contact Management**: Channel checks

**Communicative Function Categories (within each dimension):**

- Information-providing functions: Inform, Agreement, Disagreement, Answer,
  Confirm, Disconfirm
- Information-seeking functions: Question, Set-Question, Choice-Question,
  Check-Question, Propositional-Question
- Action-discussion functions: Offer, Promise, Request, Instruct, Suggest,
  Address-Request
- Feedback functions: Positive/Negative Auto/Allo-Feedback

Each dialogue act has formally defined **information-state update semantics** --
what changes it makes to the participants' shared information state.

### 7c. Clark and Brennan: Grounding in Communication

**Paper:** Clark, H.H. and Brennan, S.E. (1991). "Grounding in
Communication." In *Perspectives on Socially Shared Cognition*, APA.

**Source:** [Stanford PDF](https://web.stanford.edu/~clark/1990s/Clark,%20H.H.%20_%20Brennan,%20S.E.%20_Grounding%20in%20communication_%201991.pdf)

### Core Concepts

**Common Ground**: The collection of mutual knowledge, mutual beliefs, and
mutual assumptions that participants rely upon during interaction.

**Types of Common Ground:**

- **Communal Common Ground**: Shared by virtue of community membership
  (language conventions, cultural knowledge, professional expertise).
- **Personal Common Ground**: Built through direct interaction history with
  a specific individual (shared experiences, prior conversations).
- **Local Common Ground**: Information from the current exchange.

**The Grounding Process:**

Communication requires coordinating both **content** (what is being
communicated) and **process** (how the communication unfolds). Grounding
is the process by which participants establish mutual understanding.

**Grounding Criterion**: Participants proceed when they have sufficient
evidence that their partner has understood well enough for current purposes.

**Evidence of Understanding (Grounding Techniques):**

1. **Continued Attention**: Eye contact, posture (signals "I'm following").
2. **Relevant Next Turn**: Responding appropriately (strongest evidence).
3. **Acknowledgment**: Back-channel signals ("uh-huh", "yeah", nods).
4. **Demonstration**: Paraphrasing or restating in own words.
5. **Display**: Verbatim or partial repetition.

**Presentation-Acceptance Model:**

Each contribution to discourse has two phases:
1. **Presentation**: Speaker A presents an utterance.
2. **Acceptance**: Addressee B provides evidence of understanding.

Only after acceptance does the content become part of common ground.

### Reusability Assessment

Dialog act taxonomies provide a ready-made classification for utterance types
in our ontology. Rather than treating all utterances as plain text, we can
type them with communicative function. The grounding framework is essential for
modeling "what has been mutually established" vs. "what was merely said" --
this maps directly to the distinction between raw episodic memory and
validated common ground. Clark's presentation-acceptance model suggests that
our Commitment class should track whether commitments have been grounded
(accepted by both parties).

## 8. Time and Event Ontologies

### 8a. Allen's Interval Algebra (1983)

**Paper:** Allen, James F. (1983). "Maintaining Knowledge about Temporal
Intervals." *Communications of the ACM*, 26(11):832-843.

**Source:** [Wikipedia](https://en.wikipedia.org/wiki/Allen's_interval_algebra)

Allen's interval algebra defines **13 jointly exhaustive and pairwise disjoint
binary relations** between time intervals:

| Relation | Symbol | Inverse | Inverse Symbol |
|----------|--------|---------|---------------|
| X before Y | b | Y after X | bi |
| X meets Y | m | Y met-by X | mi |
| X overlaps Y | o | Y overlapped-by X | oi |
| X starts Y | s | Y started-by X | si |
| X during Y | d | Y contains X | di |
| X finishes Y | f | Y finished-by X | fi |
| X equals Y | = | (self-inverse) | = |

Properties:
- **Distinct**: No pair of intervals can be related by more than one type.
- **Exhaustive**: All pairs of intervals are described by exactly one type.
- **Qualitative**: No numeric time values are required.

Allen also provides a **composition table** enabling transitive reasoning:
if we know the relation between A and B, and between B and C, we can compute
possible relations between A and C.

### 8b. OWL-Time (W3C Recommendation)

**Standard:** Cox, S. and Little, C. (2017/2020). "Time Ontology in OWL."
W3C Recommendation.

**Source:** [W3C OWL-Time](https://www.w3.org/TR/owl-time/)

OWL-Time is an OWL 2 DL ontology providing vocabulary for temporal concepts.

**Core Classes:**

- `time:TemporalEntity` -- abstract superclass
  - `time:Instant` -- a temporal entity with no extent (point in time)
  - `time:Interval` -- a temporal entity with extent
    - `time:ProperInterval` -- an interval where beginning and end are
      distinct (disjoint from `Instant`)

- `time:TemporalDuration` -- duration of a temporal entity
  - `time:Duration` -- expressed using xsd:duration
  - `time:DurationDescription` -- described using calendar/clock components

- `time:TemporalPosition` -- position on a timeline
  - `time:TimePosition` -- numeric or nominal position
  - `time:GeneralDateTimeDescription` -- calendar-based position

- `time:DateTimeInterval` -- a ProperInterval corresponding to a single
  calendar/clock element (year, month, day, etc.)

**Key Properties:**

- `time:hasBeginning` / `time:hasEnd` (TemporalEntity to Instant)
- `time:hasTemporalDuration` (TemporalEntity to TemporalDuration)
- `time:hasTime` (generic attachment of temporal info to any resource)
- `time:inXSDDateTime` / `time:inXSDDate` (position as xsd value)

**Allen Relations as OWL Properties:**

OWL-Time implements all 13 Allen relations as object properties:
`time:intervalBefore`, `time:intervalMeets`, `time:intervalOverlaps`,
`time:intervalStarts`, `time:intervalDuring`, `time:intervalFinishes`,
`time:intervalEquals`, plus their inverses (`time:intervalAfter`,
`time:intervalMetBy`, `time:intervalOverlappedBy`, `time:intervalStartedBy`,
`time:intervalContains`, `time:intervalFinishedBy`).

Extensions add: `time:equals`, `time:hasInside`, `time:disjoint`,
`time:notDisjoint`.

### 8c. Event Calculus

**Paper:** Kowalski, R. and Sergot, M. (1986). "A Logic-Based Calculus of
Events." *New Generation Computing*, 4(1):67-95.

**Source:** [Springer](https://link.springer.com/article/10.1007/BF03037383) |
[Shanahan, "The Event Calculus Explained"](https://www.doc.ic.ac.uk/~mpsha/ECExplained.pdf)

The Event Calculus is a formalism for reasoning about events and their
effects over time. It is characterized by reifying both time and events.

**Core Predicates:**

- `Happens(e, t)` -- event e occurs at timepoint t
- `HoldsAt(f, t)` -- fluent (time-varying property) f is true at time t
- `Initiates(e, f, t)` -- if event e occurs at t, fluent f becomes true
  after t
- `Terminates(e, f, t)` -- if event e occurs at t, fluent f becomes false
  after t
- `Clipped(t1, f, t2)` -- fluent f is terminated between t1 and t2
- `InitiallyP(f)` -- fluent f holds at time 0

**Key Axiom (Inertia / Frame Axiom):**

A fluent initiated by an event continues to hold by default until some
event terminates it. This is the "common sense law of inertia" -- things
stay the same unless something changes them.

**Relevance to Agent Memory:**

The Event Calculus provides a formal framework for:
- Tracking state changes caused by events (e.g., a commitment being created,
  modified, or fulfilled)
- Reasoning about what holds at a given time based on the history of events
- The inertia axiom models "persistence" of beliefs and states

### Ontological Concepts from Temporal Frameworks

| Concept | Proposed Class | Source | Relations |
|---------|---------------|--------|-----------|
| Temporal Entity | `time:TemporalEntity` | OWL-Time | superclass for Instant, Interval |
| Instant | `time:Instant` | OWL-Time | `time:inXSDDateTime` |
| Interval | `time:ProperInterval` | OWL-Time | `time:hasBeginning`, `time:hasEnd` |
| Duration | `time:Duration` | OWL-Time | xsd:duration value |
| Allen Relation | (13 object properties) | OWL-Time/Allen | between ProperIntervals |
| Event | `Event` | EC/General | `happensAt` Instant, `hasDuration` |
| Fluent | `Fluent` | Event Calculus | `holdsAt` time, `initiatedBy` Event |
| State Change | `StateChange` | Event Calculus | `initiates`/`terminates` Fluent |

### Reusability Assessment

**OWL-Time is directly importable** as the temporal backbone of our ontology.
It provides everything needed: instants, intervals, durations, positions, and
Allen relations. Allen's interval algebra (already implemented in OWL-Time)
gives us qualitative temporal reasoning without requiring exact timestamps.

The Event Calculus concepts (fluents, initiation, termination) are useful for
modeling how agent beliefs and commitments change over time, but they are
better implemented as design patterns (using OWL-Time temporal entities) than
as direct imports, since the Event Calculus is formulated in first-order logic
rather than OWL.

```
        |
        |-- ProceduralMemory
              |-- ProductionRules (ACT-R, Soar)

              |-- ToolPolicies (agent-specific)
```

```
AgentCycle (= CoALA decision loop = Soar decision cycle = LIDA cognitive cycle)
  |-- PerceptionPhase

  |-- ReasoningPhase

  |     |-- Deliberation (BDI)
  |     |-- Reflection (Generative Agents)
  |
  |-- ActionPhase

  |
  |-- LearningPhase
        |-- Consolidation (episodic -> semantic)
```

```
MentalState
  |-- Belief (informational)
  |     |-- DispositionalBelief (standing knowledge)

  |-- Desire (motivational)
```

```
Conversation
  |-- Turn
```

```
time:TemporalEntity  (import OWL-Time)
```
