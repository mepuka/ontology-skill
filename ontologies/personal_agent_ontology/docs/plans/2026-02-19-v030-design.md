## 1. Compaction Preservation/Loss Trace Model

**Problem**: UC-006 asks "what was preserved" during compaction, but only
`producedSummary` exists. No way to query preserved vs dropped items.

**Approach**: Hybrid PROV-O + lightweight domain extension.

CompactionEvent already inherits from `prov:Activity`. We lean on PROV-O's
native vocabulary for the core pattern and add a domain-specific n-ary
reification for per-item metadata.

### Core Pattern (PROV-O native)

- `prov:used` — CompactionEvent used (consumed) items as input
- `prov:wasInvalidatedBy` — dropped items were invalidated by compaction
- `prov:wasDerivedFrom` — summary was derived from preserved items

### Domain Extension

**New classes (2)**:
- `CompactionDisposition` (GDC, BFO_0000031) — n-ary reification linking
  event, item, and fate with optional rationale
- `ItemFate` (subclass of Status) — controlled vocabulary for compaction outcomes

**New individuals (4, ItemFate)**:
- `Preserved` — item content incorporated into summary
- `Dropped` — item excluded from summary
- `Summarized` — item partially represented in summary
- `Archived` — item moved to long-term storage

**New properties (6)**:

| Property | Type | Domain | Range | Characteristics |
|----------|------|--------|-------|-----------------|
| compactedItem | obj | CompactionEvent | prov:Entity | subPropertyOf prov:used |
| hasCompactionDisposition | obj | CompactionEvent | CompactionDisposition | — |
| dispositionOf | obj | CompactionDisposition | prov:Entity | functional |
| hasItemFate | obj | CompactionDisposition | ItemFate | functional |
| fateReason | data | CompactionDisposition | xsd:string | — |

**PROV imports to add**: `prov:used`, `prov:wasInvalidatedBy`,
`prov:Derivation`, `prov:qualifiedDerivation`, `prov:entity`, `prov:hadActivity`

**CQ updates**:
- Revise CQ-010 to query preserved vs dropped items
- Add CQ-052: "What items were dropped during a compaction event?"
- Add CQ-053: "What fraction of items were preserved during compaction?"

## 2. Eviction Eligibility and Last-Access Semantics

**Problem**: UC-011 asks for tier + last access + eviction eligibility.
Current model captures tier and operation history but not eviction semantics.

**New properties (2)**:

| Property | Type | Domain | Range | Characteristics |
|----------|------|--------|-------|-----------------|
| hasLastAccessTime | data | MemoryItem | xsd:dateTime | functional |
| isEvictionCandidate | data | MemoryItem | xsd:boolean | functional |

**New CQs**:
- CQ-054: "When was a memory item last accessed?"
- CQ-055: "Which memory items in a given tier are eligible for eviction?"

**SHACL**: Update MemoryItemShape — allow `hasLastAccessTime` (0..1).

**ABox**: Add `hasLastAccessTime` to existing memory items. Mark at least
one item with `isEvictionCandidate true`.

## 3. Cross-Session Resume Linkage

**Problem**: UC-016 "resume multi-session work" has no explicit
session-to-session linking for pending goals.

**New properties (2, inverse pair)**:

| Property | Type | Domain | Range | Characteristics |
|----------|------|--------|-------|-----------------|
| continuedFrom | obj | Session | Session | functional |
| continuedBy | obj | Session | Session | inverse of continuedFrom |

**Constraint**: A session can continue from at most one prior session
(functional). A session can be continued by multiple sessions (one-to-many).

**New CQ**:
- CQ-056: "Which session continued from a given prior session?"

**ABox**: Link two existing sessions as a continuation chain.

## 4. HasKey Identity Contracts (DN-05)

**Problem**: Identity-key strategy deferred as DN-05. Implementation teams
must invent identity contracts without ontology guidance.

**New data properties (3)**:

| Property | Type | Domain | Range | Characteristics |
|----------|------|--------|-------|-----------------|
| hasAgentId | data | AIAgent | xsd:string | functional |
| hasSessionId | data | Session | xsd:string | functional |
| hasConversationId | data | Conversation | xsd:string | functional |

**New axioms**: `owl:HasKey` on each class with its ID property.

pao:Session owl:hasKey ( pao:hasSessionId ) .
pao:Conversation owl:hasKey ( pao:hasConversationId ) .

**New CQ**:
- CQ-057: "What is the unique identifier for a given agent/session/conversation?"

**ABox**: Add ID string values to existing individuals.

**Design note**: Resolve DN-05 in axiom-plan.yaml.

## 5. Lifecycle Transition Semantics (State Transition ODP)

**Problem**: Model has current status values but no transition history.
No way to query previous states or the sequence of changes.

**Approach**: Reified Event pattern — StatusTransition as subclass of Event
(BFO:Process). Based on PackML State Machine ODP and W3C N-ary Relations.

### New Classes (3)

| Class | Parent | BFO | Module |
|-------|--------|-----|--------|
| StatusTransition | Event | Process (BFO_0000015) | pao-event |
| SessionStatusTransition | StatusTransition | — | pao-event |
| TaskStatusTransition | StatusTransition | — | pao-event |

### New Properties (6)

| Property | Type | Domain | Range | Characteristics |
|----------|------|--------|-------|-----------------|
| fromStatus | obj | StatusTransition | Status | functional |
| toStatus | obj | StatusTransition | Status | functional |
| transitionSubject | obj | StatusTransition | owl:Thing | functional |
| triggeredBy | obj | StatusTransition | Event | — |
| previousTransition | obj | StatusTransition | StatusTransition | functional |
| nextTransition | obj | StatusTransition | StatusTransition | functional, inverse of previousTransition |

### Design Decisions

- Keep `hasStatus` as the current-state shortcut (functional property). StatusTransition provides the audit trail.
- Do NOT make previousTransition/nextTransition transitive. Use SPARQL property paths for chain queries.
- Add SHACL SPARQL constraint for chain consistency: `toStatus(N) == fromStatus(N+1)`.

### New CQs

- CQ-058: "What status transitions has a task undergone?"
- CQ-059: "What was the previous status of a session before it ended?"
- CQ-060: "What event triggered a given status transition?"

### ABox

Sample transition chains:
- Task: Pending → InProgress → Completed (2 transitions)
- Session: Active → Ended (1 transition)

### SHACL

- StatusTransitionShape: fromStatus (1..1), toStatus (1..1), transitionSubject (1..1)
- Chain consistency SPARQL constraint
